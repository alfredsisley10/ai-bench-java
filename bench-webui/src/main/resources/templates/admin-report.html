1. What is being benchmarked

ai-bench evaluates how well large language models (LLMs) can autonomously diagnose and fix bugs in a realistic enterprise-style Java codebase (banking-app). Each bug is a verified regression: a known commit that breaks at least one pre-existing test case. The model is given a problem statement plus a configurable amount of code context, and is asked to produce a git-style patch. The patch is applied to a clean worktree, the Gradle test suite is executed, and the run is graded by counting the JUnit-XML test outcomes.

Unlike toy benchmarks that score by free-form output similarity, ai-bench scores by whether the fix actually compiles and passes the same tests a human reviewer would require.

2. Methodology

Context providers

  • Oracle — for calibration only. The model is handed the exact files the human fix touched. Oracle entries are excluded from the leaderboard since they leak ground truth.
  • BM25 — classic lexical retrieval over the repository using the bug's problem statement as the query. No runtime traces, no LLM in the retrieval step.
  • AppMap — uses pre-recorded runtime traces (AppMap files generated from the project's own unit tests) ranked via appmap search (vector retrieval, no LLM in retrieval). Captures behavior the source alone doesn't show.

AppMap modes

  • OFF — no traces injected into the prompt.
  • ON — top-K traces selected by the active context provider are appended to the prompt as compact summaries.

Scoring rubric

A run is PASSED only if (a) the produced patch applies cleanly with git apply, (b) the project compiles, and (c) every previously-failing test for the bug now passes and no previously-passing tests regress.

4. Pre-training contamination warning

The runs below passed with contextProvider = none — no source code, no traces, no oracle file list was supplied to the model. A passing patch in this configuration means one of two things:

  1. Model contamination — the bug (or its real fix commit) was in the model's pre-training data. The model is effectively answering from memory; benchmark scores against this model on this bug are not meaningful.
  2. Bug-definition contamination — the problem statement or hints below give the fix away. Any reasonably- capable model would pass without code, with or without pre-training exposure.

To verify which case applies, read each problem statement / hint set below and ask: "could a competent engineer write the fix from this text alone, with no access to the codebase?" If yes, the bug definition is the leak. If no, the model is.

6. Glossary

BM25
Classic information-retrieval ranking function (Okapi BM25). Used here to surface candidate files from the repository.
AppMap
Runtime trace format (recorded from JUnit tests) capturing executed methods, SQL, HTTP calls and code paths.
Oracle context
The exact files the human fix touched. A control / upper bound — not eligible for the leaderboard.
Pass rate
Fraction of attempts that produced a verified passing patch.
Avg cost
Mean estimated USD cost per successful run, computed from the provider's published per-token pricing.
Pre-training contamination
A model passing a benchmark with no context provider supplied cannot be reasoning from the codebase — it is either reproducing a fix from its training corpus, or the prompt itself is leaky enough to give the answer away. Such (model, bug) pairs are flagged in §4 and excluded from interpretation of headline scores until the leak source is identified.
