Runtime options

Picks how the harness chooses which source files to ship in the prompt. Does not change which LLM is called.

  • None: problem statement only — measures how much the model can fix without seeing any code (worst- case baseline). Also doubles as a training-data contamination check: if the model produces a correct patch from the prompt alone, it likely saw this bug (or its fix) during pre-training, which invalidates the benchmark for that model.
  • Oracle: hand-curated filesTouched list from the bug's YAML — guarantees the right file is in the prompt (best-case ceiling).
  • BM25: TF-IDF retrieval over every *.java in banking-app, ranked by terms drawn from the problem statement; top 5 files. No ground-truth leak — closest to a real-world repo search workflow.
  • Agentic / BM25 + Agentic: the LLM iteratively requests files via tool-use. Both are preview-only today — selecting them falls back to Oracle and writes a fallback note on the audit page.
  • AppMap Navie: Navie's AppMap-driven retrieval, configured on the AppMap setup page.

AppMap traces provide runtime context (call graphs, data flows, HTTP interactions) to the LLM solver. ON_RECOMMENDED is the default and balances trace quality with performance. ON_ALL gives maximum context but increases build time and token usage.

Number of independent solver attempts per bug. Multiple seeds measure consistency (pass@k). Each seed uses a different random seed for the LLM, producing varied solutions for the same prompt. Typical: 3 for quick runs, 5-10 for statistical significance.
