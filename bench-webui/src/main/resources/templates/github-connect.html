Pick whichever auth flow your environment supports. Option 1 (device flow) avoids ever holding a long-lived secret on this host — best for shared / unattended machines and orgs that mandate OAuth Apps. Option 2 (personal access token) is faster to set up if you already have a PAT or your org doesn't allow OAuth Apps.

Option 1 — Device flow (OAuth App)

Uses GitHub's device-authorization grant: this WebUI generates a short user code, you paste it into a browser tab on any device, and GitHub returns an access token scoped to whatever the OAuth App is configured for. No long-lived secret ever sits on this host — the token lives in your session and is wiped on restart.

How to set this up (one-time)
  1. Create an OAuth App on the host you configured above:
    • github.com: Settings → Developer settings → OAuth Apps → New OAuth App (or your org's OAuth Apps page if the OAuth App should be org-owned)
    • GitHub Enterprise Server: then OAuth Apps → New OAuth App
  2. Fill in the form:
    • Application name: anything (e.g. ai-bench-webui)
    • Homepage URL: http://localhost:7777/
    • Authorization callback URL: http://localhost:7777/github/callback (not used by device flow but required by the form)
    • Check Enable Device Flow ⚠️ (this is required and OFF by default — easy to miss)
  3. After saving, GitHub shows a Client ID at the top of the app page (looks like Ov23li… on github.com or Iv1.… on older OAuth Apps). Copy it.
  4. Paste the Client ID below and click Start device flow. The page returns a verification URL + a short user code.
  5. Open the verification URL in any browser, paste the user code, and approve. The WebUI polls for the resulting access token and saves it to your session.

Required scopes: typically repo (issues, PRs, contents) and read:org (membership info). Configure these on the OAuth App page if needed for your benchmarks.

The verification URL returned by this step will be on the host you configured above ().

Option 2 — Personal access token (PAT)

Skip the OAuth dance and just paste a token. GitHub supports two PAT flavors: classic (ghp_…, scopes-only, lives forever unless you set an expiry) and fine-grained (github_pat_…, scoped to specific repos with individual permissions, mandatory expiry). Both work here — ai-bench picks whichever is on the Authorization header.

How to create a PAT

Classic PAT (simplest, recommended for dev/test):

  1. Go to (enterprise host) github.com/settings/tokens
  2. Click Generate new token (classic)
  3. Set a note (e.g. ai-bench-webui), pick an expiry, and select scopes:
    • repo — full repo access (required for issue / PR / contents read)
    • read:org — read org membership (required if you list org repos)
    • workflow — only if you also need to read GitHub Actions runs
  4. Click Generate token, copy the resulting ghp_… string immediately (GitHub only shows it once)

Fine-grained PAT (recommended if your org enforces them):

  1. Go to github.com/settings/personal-access-tokens
  2. Click Generate new token, pick the resource owner (your account or the org)
  3. Choose Repository access (specific repos or all)
  4. Under Repository permissions grant at minimum: Contents: Read, Issues: Read, Pull requests: Read, Metadata: Read
  5. Generate, copy the resulting github_pat_… string

Storage & security: ai-bench keeps the token in your session only — never written to disk by the WebUI. The session ends when you close the browser / restart bench-webui. To persist across restarts, set the GITHUB_TOKEN env var when launching ./gradlew bootRun.

Token format is not enforced — public github.com issues ghp_ (classic) and github_pat_ (fine-grained) prefixes, but self-hosted GitHub Enterprise deployments may mint tokens with different prefixes. Validation happens when you Test GitHub access.
